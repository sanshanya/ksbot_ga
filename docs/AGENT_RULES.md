# Agent Project Rules

编码前创建、结束后删除 `.agent/TASK`：

```text
OUTCOME: 可观察结果
NON-GOALS: 不做什么
VERIFY: 实际验证
ADDS: 新增永久概念，或 none
```

无法明确 `OUTCOME` 与 `VERIFY` 时不得编码。

## 实现

优先级：删除 → 复用 → 修改 → 新增局部代码 → 新增永久概念。禁止假想扩展、兼容层、备用路径、单实现抽象和新旧并存；替换时同步删除旧代码、测试、配置和文档。

同一行为只保留一条路径。拆分必须形成可独立理解的职责并降低任务切片，不能只移动代码或规避预算。禁止压行、把逻辑搬进 Prompt、顺手重构和提交临时脚本/报告。

覆盖、绕过或削弱 GenericAgent 原生行为前必须更新 `PROJECT.md`；削弱核心能力须获用户明确批准。

## 文档、测试与完成

`PROJECT.md` 是唯一长期设计文档，只保留当前边界、索引、契约、验证和限制；历史证据进入 Release、commit 或运行审计。

一个错误行为只保留一个最强测试；等价输入参数化；避免断言私有容器、精确文案和无产品意义的中间值。并发、审批后原调用单次执行、事件去重、附件证据和 GA 原生能力必须保留。Mock 不能替代真实 WPS、模型和 Kubernetes 验收。

阶段提交前执行 `VERIFY`、pytest、ruff、token budget 和 diff 检查；压缩代码/测试/文档，确认无临时内容、双路径和架构阻断项。减负后的总量与任务切片都是 ratchet；增长必须对应批准的新产品契约。
