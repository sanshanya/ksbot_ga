# Agent Project Rules

## 1. 任务入口

开始修改前，创建 `.agent/TASK`。该文件必须被 `.gitignore` 忽略，任务结束后删除。

```text
OUTCOME: <完成后可观察到的结果>
NON-GOALS: <明确不做的内容>
VERIFY: <实际验收命令或步骤>
ADDS: <新增永久文件/API/配置/依赖/运行模式，或 none>
```

无法明确 `OUTCOME` 或 `VERIFY` 时，不得编码。

## 2. 最小实现

按以下顺序解决问题：

```text
复用现有代码 → 修改现有代码 → 新增局部代码 → 新增永久概念
```

永久概念包括文件、公共 API、配置、依赖、抽象层和运行模式。

只有当前需求无法通过现有结构完成时，才允许新增，并必须提前写入 `ADDS`。

禁止为假想未来增加通用化、扩展点、兼容层、配置项或备用路径。

多个方案都正确时，选择文件更少、概念更少、路径更短、代码更少的方案。

## 3. 单一路径

同一行为只保留一种实现。

新实现替代旧实现时，在同一阶段删除旧代码、旧配置、旧测试和旧文档。

默认禁止：

* 新旧实现并存；
* wrapper 套 wrapper；
* legacy fallback；
* 单实现 interface、factory 或 strategy；
* 只有一个调用者的公共抽象；
* 仅为一次任务创建 helper、模块或框架。

只使用一次且没有独立语义的代码应直接内联。

## 4. 严格范围与压缩

只修改实现 `OUTCOME` 所必需的文件。

禁止顺便重构、统一风格、改名或修复无关问题。

验证通过后，提交前重新检查整个 diff：

```text
能否删除？
能否复用现有实现？
能否合并分支？
能否内联单用途抽象？
能否删除配置、兼容和中间层？
```

删除后仍能通过 `VERIFY` 的内容必须删除。

临时脚本、调试日志、探针、生成数据、任务总结和实现报告不得进入仓库。

## 5. 项目文档

仓库只保留一个长期设计与实现文档：

```text
PROJECT.md
```

`PROJECT.md` 只描述当前真实状态：

* 项目目标与边界；
* 核心设计；
* 主要模块及职责；
* 关键数据流；
* 外部接口与依赖；
* 运行、配置和验证方式；
* 必须长期保留的设计约束。

禁止写入：

* 开发过程；
* commit 记录；
* 已废弃设计；
* 临时计划；
* 重复源码内容；
* 可以直接从代码推导的细节；
* 大段背景、解释和示例。

每次阶段提交前必须检查并更新 `PROJECT.md`：

* 设计或实现发生变化时，修改对应内容；
* 删除已经失效的内容；
* 合并重复描述；
* 文档未受影响时，不做形式化修改。

`PROJECT.md` 必须始终描述当前代码，而不是项目历史。

新增内容后必须同时压缩全文。能够删除而不影响后续 Agent 理解项目的内容必须删除。

除 `PROJECT.md`、用户接口文档和必要部署文档外，默认不得新增长期文档。

## 6. 测试

先运行现有测试。只有现有测试无法拒绝本次目标错误时，才新增测试。

每个新增或实质修改的测试必须紧邻以下单行注释：

```text
TEST-CONTRACT: req=<需求/故障> | rejects=<错误行为> | gap=<现有缺口> | revert=<移除哪项修复后失败> | mock=<none或边界=>真实验证>
```

规则：

* 一个错误行为只保留一个测试；
* 等价输入使用参数化；
* 核心修改路径不得 mock；
* Agent 自己新增的测试不能单独证明任务完成；
* 涉及 WPS、真实模型、Kubernetes 或其他外部链路时，Mock 不能替代至少一次目标环境验收；
* 当前环境无法执行真实验收时，必须明确标记“未验证”，不得把测试通过写成真实闭环通过；
* 禁止删除、跳过、弱化或绕过现有测试和 CI。

测试通过但 `OUTCOME` 未被真实验证，任务仍未完成。

## 7. 阶段完成

开发过程使用工作区或临时快照，不为每个函数和修复创建永久 commit。

只有形成稳定阶段时才提交。稳定阶段必须：

```text
在声明范围内可运行
有明确且实际执行过的验证方式
可继续开发
无临时代码或新旧双路径
无已知会推翻当前架构的缺陷
未完成能力与已知限制已准确记录
```

提交标题只描述阶段结果，不记录开发过程。

阶段提交前依次执行：

1. 执行 `VERIFY`；
2. 运行 `scripts/token_budget.py`，确认每个单模块不超过 30000 token，并在阶段说明中记录受影响模块的当前 token 数；不提交生成报告；
3. 压缩代码和测试 diff；
4. 更新并压缩 `PROJECT.md`；
5. 通过 Harness 检查；
6. 创建阶段提交；
7. 删除 `.agent/TASK`。

Agent 不得自行降低规则、测试或验收条件。最终是否完成由 Harness 判定。
